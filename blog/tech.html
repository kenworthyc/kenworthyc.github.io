Regular Expressions: to regex or not to regex?

12.12.2015

what are regular expressions?

Conceptually, regular expressions derive partially from the study of computers, mathematics, and language: as you might intuit, they're an approximation of a word or a phrase. Specifically, regular expressions are approximations of a string with the capacity for complex parameters in searching through and comparing strings of text.

Practically, regexes are useful tools to match patterns within a given text input. They're almost indispensible because of their flexibility in finding patterns which are similar, not merely identical to, their set patterns. Regexes harness this capability by pairing its searching pattern (e.g. the precise string) with modifiers that tersely replace a long, annoying if-else chain (for every uppercase and lowercase character, say). They're found in many programming languages, and in JavaScript, they look like this:

var regex = /ExAmPlE/i
// literal constructor = /pattern to search in text/modifier
// (i = case-insensitive)

The library of specificities you can set for your defined pattern is impressive, and w3Schools has a good starter list to get an idea of the kinds of patterns you can search for, as well as the modifiers on those patterns you can use.
Note: these are JavaScript Regexp docs, and different languages may have slightly different applications or syntactical parameters. For contrast, check out the Ruby Regexp docs.

how about a popular regular expression?

Regular expressions are quite useful in the management of a contemporary website; if the site is at all interactive, it (literally) invites the user to play with it through their mouse and keyboard— and those pesky keyboards are notoriously inaccurate (although it could be user error). For instance, to get an email address, and to be sure the address typed in is a valid, regular expressions to the rescue! A regex can scan and confirm an email is valid in one line:

Let's break it down.

  1. The regex signals the start and end of input, which correspond to the characters ^ and $, respectively. Next, the range (marked with brackets— almost like an array, no?) of possible characters in an email is A-Z, 0-9, and ._%+-.
  2. Next, outside the brackets, is a mandatory character— the at-sign noted by +@. Then there's another range of possible letters, numbers, and a period and dash, all of which are valid website addresses for an email host.
  3. Then there's another mandatory character outside the brackets: the period before a domain suffix. It looks like the way the at sign was added, but with one extra. +\. adds the period with an escape character. The escape character (\) is crucial! necessary! not optional!, since a period in a regex is a wildcard character (like *.doc for selecting all the docs in a folder from the command line).
  4. Finally, there's the letter range for the domain, which can only be letters (like .com, .sy, or .info), and must be no less than two characters: hence, [A-Z]{2,}. The curly braces define the possible length of a string to match in the regex.
Note: the author of this example doesn't include lowercase letter characters explicitly, as there's an option on the 'regex engine' (which compiles the regex code) for case-insensitivity, which can be flipped on. However, there's a way you can explicitly include lowercases— how do you think it might work?

bonus: regex vs. regexp?

Which is the 'correct' abbreviation? As this entertainingly incensed StackOverflow thread demonstrates, both are valid, although 'regex' is slightly more common. You'd be easily understood using either spelling.
