Functional vs. Object-Oriented Programming

12.6.2015

At the beginning of this week, I was asked to identify two similarities and two differences between Ruby and Javascript. I picked out a few syntax changes, then it occurred to me (having just learned about JavaScript objects)— they're both Object-Oriented!
Well, turns out... nope.

object-oriented programming, oop!

In simple terms, object-oriented programming languages (OOPLs) are constructed of (and can construct) self-contained blocks of code which have collections of properties. In such a language, that's all an object is: a collection of properties (or attributes, or adjectives, if you prefer). Object-oriented languages are built of objects, and let the programmer build new objects.

Ruby is a great example of an OOPL: all of its interactive parts, like strings, numbers, methods, and classes, have a shared ancestor: the Object class. (And Class is a class too, but the purpose of any Class in Ruby is to create and define the properties of some new Object. (That's why the constructor looks like ClassName.new()— it's building a new instance, a new individual object, from the blueprint of ClassName.)

JavaScript has its own OOPL-like traits: one of the main tools for a JavaScript programmer is (ruby's Hash-like) Object. In addition, functions in JavaScript fulfill the description above: 'self-contained blocks of code which have collections of properties'. OOP can be an interesting and user-friendly way to conceive a programming language; there is a vast freedom the new programmer has over the actions and tasks to-be (assuming they get far enough to experiment with constructors, classes, and objects).

functional programming!

Functional programming languages, in opposition to OOPLs, assume a basic set of methods, input types, objects, and the like. Most importantly, they assume a global (although not entire) knowledge of available functions and their quirks. One purpose of any programming language is to allow the programmer to intuitively achieve a level of 'abstraction'; that is to say, the language can condense and express what the computer has to do in the program easily and simply.

As shown in the example below, a JavaScript programmer can simplify and condense the longer syntax of a for loop (which affects each element within the array it's passed) into the function forEach. Voilà: functional programming makes basic iterators simpler to type in.

In point of fact, JavaScript is a hybridized programming language that incorporates certain flexibilities of Object-Oriented Programming with the prescriptive nature of Functional Programming. I'm not nearly knowledgeable enough to say with certainty the dividing line between the two, but for now... I'll have to think of it as the best of both worlds.
